# ADR-078: Add Paradise Public CLI And Reusable Verification Action

## Status
Accepted

## Date
2026-05-25

## Context
Paradise has a strong internal GARM/GEWC runtime surface, but first-time users
should not need to start the API server or learn every internal runtime term
before seeing useful evidence.

The public repository needs a minimal path that:

- runs locally without sockets, network access, API keys or model training;
- generates visible evidence files;
- preserves `claim_allowed=false` and `agi_claim=false`;
- makes dry-run and permission-first behavior obvious;
- can be reused by GitHub Actions.

## Decision
Add a public `paradise` binary with three commands:

- `paradise status`;
- `paradise worldcell`;
- `paradise run --dry-run <intent>`.

Add `make paradise-quickstart` and a reusable composite action at
`.github/actions/paradise-verify`.

Keep `eden-garm` and `edenctl` as advanced runtime/API tools. The public CLI is
not a second runtime; it is a socket-free front door over the existing runtime
modules and evidence contracts.

## Alternatives Considered

### Keep only `eden-garm`

This preserved one runtime entry point but forced new users into REPL or daemon
flows before they could understand the product surface.

### Keep only `edenctl`

This fit the live API model but required a running socket server for the first
experience.

### Build a separate demo script

This would be faster, but it would create another public surface outside the
Rust workspace and make tests weaker.

## Consequences

- Public onboarding starts from Paradise terminology.
- Advanced GARM/GEWC surfaces remain intact.
- Quickstart evidence is generated by Rust code and can be tested.
- The reusable Action verifies the public path without granting autonomous tool
  authority.
- Future public commands should preserve the same dry-run and evidence-first
  contract.
