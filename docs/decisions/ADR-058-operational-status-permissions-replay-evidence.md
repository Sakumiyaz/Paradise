# ADR-058: Operational Status, Persistent Permissions And Replay Evidence

## Status
Accepted

## Date
2026-05-24

## Context
The runtime had generated operational contracts and black-box gates, but the
external surface still needed four stronger properties:

- a live status document that distinguishes ready and degraded operation;
- a persistent permission file used by GEWC before mutating actions;
- a replay API that lets operators inspect why a decision was taken;
- a compact evidence bundle for release review.

Without those properties, operational validation was spread across multiple
artifacts and was harder to inspect from outside the running process.

## Decision
Add operational control endpoints and artifacts:

- `/api/operational/status` reports readiness, degraded state, handler state,
  latest decision evidence and permission summary.
- `/api/operational/permissions` serves `operational_permissions.json`, the
  persistent permission boundary generated by `operational api eval`.
- `/api/operational/replay` lists recorded GEWC decisions.
- `/api/operational/replay?decision_id=<id>` returns the matching decision and
  completion evidence without re-executing it.
- `make operational-evidence-bundle` writes
  `operational_evidence_bundle.json` from the generated state directory and
  runtime logs.

GEWC now checks the persistent permission boundary before execution. Remote
network action remains denied by default unless the explicit runtime override is
enabled.

## Alternatives Considered

### Keep status as static documentation only

This would be simpler, but external consumers would not be able to inspect the
live runtime state from the same API surface used for operations.

### Treat replay as re-execution

Re-execution would be more powerful but unsafe for an operational evidence API.
The current replay surface is deliberately read-only and reconstructs recorded
decisions from runtime evidence.

### Hard-code permissions only in command handlers

Handler-local checks are useful, but a persistent permission artifact makes the
boundary auditable, stable across processes and visible to dry-run consumers.

## Consequences

- External validation can inspect operational status, permissions and replay
  without private runtime access.
- The black-box gate now produces a bundled evidence manifest in addition to
  individual artifacts.
- Permission decisions become easier to audit because dry-runs and GEWC execution
  reference the same permission keys.
- These changes remain operational evidence only and keep `claim_allowed=false`
  and `agi_claim=false`.
