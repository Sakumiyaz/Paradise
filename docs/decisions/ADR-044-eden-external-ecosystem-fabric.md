# ADR-044: Eden External Ecosystem Fabric

## Status

Accepted

## Date

2026-05-22

## Context

OpenCog Hyperon has a real external ecosystem advantage: public repositories, documentation, papers, public community channels and a long research lineage. EDEN should acknowledge that strength without copying AtomSpace, MeTTa or metagraph rewriting, and without claiming public adoption that has not happened.

The existing EDEN architecture already had SDK, adapter and reproducible demo contracts, but that is not enough to make the external ecosystem sector deep. EDEN needs an original ecosystem model that can accept outside agents, tools, models, datasets and benchmarks while preserving GEWC authority, Praxis evidence, policy, provenance, reproducibility and no-claim boundaries.

## Decision

Add `external ecosystem eval` as the validation command for the Eden External Ecosystem Fabric.

It writes:

- `eden_external_ecosystem.json`
- `ecosystem_participation_contract.json`
- `ecosystem_interop_matrix.json`
- `ecosystem_certification_ladder.json`
- `ecosystem_onboarding_runbook.json`
- `ecosystem_governance_model.json`
- `ecosystem_benchmark_exchange.json`

The design is EDEN-native:

- external contributions become governed capability capsules, not independent cores;
- capsules route through GEWC body handlers;
- Praxis evidence binds contribution effects to intent, constraints, provenance and traces;
- certification distinguishes local smoke evidence from independent reproduction and real adoption;
- benchmark exchange accepts external task capsules without allowing unsupported capability claims.

## Alternatives Considered

### Copy Hyperon-style substrate concepts

Rejected. EDEN's path is GEWC-governed cognition in action, not AtomSpace/MeTTa/metagraph compatibility as a central identity.

### Treat SDK, adapters and demos as the whole ecosystem

Rejected. Those are necessary but too shallow. A durable ecosystem needs participation contracts, interoperability, certification, onboarding, governance and benchmark exchange.

### Claim EDEN already beats OpenCog in adoption

Rejected. EDEN can validate local ecosystem architecture readiness, but public adoption, external contributors and historical maturity are external facts that must be earned.

## Consequences

- `sovereign cognition eval` now uses the external ecosystem fabric as evidence for the Hyperon ecosystem sector.
- The release-candidate package and independent validator require the new ecosystem artifacts.
- The local held-out harness includes an external ecosystem case.
- The capability registry gains `eden_external_ecosystem` as a locally validated, no-claim architecture capability.
- EDEN's ecosystem comparison remains precise: stronger local architecture target, not a false public-adoption claim.
